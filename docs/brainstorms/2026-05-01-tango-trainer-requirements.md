# Tango Trainer (Flutter MVP)

## Problem Frame

Игрок (один пользователь — автор приложения) ежедневно решает LinkedIn Tango и хочет систематически улучшать скорость и точность распознавания дедукций. Стандартные пазл-приложения дают только сам пазл и опционально таймер: они не различают, *какие* техники медленные, и не дают точечной тренировки именно на слабые. Цель — личный тренажёр, который наблюдает за обычной игрой, диагностирует слабые именованные дедукции по converging-evidence (latency + hint + ошибки) и подаёт точечные drill-ситуации именно на слабых техниках через FSRS-планировщик.

Полностью локально, sqlite, без сервера, без облака, без рекламы, без аккаунтов.

---

## Actors

- A1. **Игрок** (single-user, владелец приложения): играет полные партии, иногда запрашивает hint, периодически проходит drill-сессии.
- A2. **Solver-движок**: для любой позиции возвращает каталог доступных дедукций, какая самая дешёвая, какие клетки она форсирует. Используется и для подсказок, и для разметки ходов, и для генерации drill, и для оценки baseline-времени.
- A3. **Diagnostic engine**: собирает per-move телеметрию, классифицирует её по эвристикам, считает per-heuristic mastery, питает FSRS-очередь.
- A4. **Drill-планировщик** (FSRS поверх per-heuristic state): решает, какие техники подавать в следующую drill-сессию.
- A5. **Генератор уровней**: по target-mix histogram возвращает уникально-решаемый puzzle с заданным составом эвристик.

## Key Flows

- F1. **Обычная партия как источник диагностики**
  - **Trigger:** A1 открывает full-game режим
  - **Actors:** A1, A2, A3
  - **Steps:**
    1. Генератор отдаёт обычный 6×6 Tango (без целевого heuristic-mix)
    2. На каждый ход A3 пишет move-event: `{heuristic_tag, latency_ms, was_correct, hint_requested, contaminated_flag}`
    3. Если ход требует дедукции вне MVP-словаря — тег `Composite(unknown)`, в drill не пойдёт, но логируется
    4. По окончании партии — diff с solver-оптимальным путём, генерация drill-карточек на 2–3 самые долгие/неоптимальные паузы → попадают в FSRS-очередь
  - **Outcome:** обновлены per-heuristic mastery + FSRS-очередь; A1 видит end-of-session summary
  - **Covered by:** R1, R2, R4, R6, R7, R12, R13

- F2. **Drill-сессия (база 10 + расширение)**
  - **Trigger:** A1 нажимает «Drill» в главном меню
  - **Actors:** A1, A4, A5, A3
  - **Steps:**
    1. A4 выбирает 10 техник из FSRS-очереди (приоритет: due + слабая mastery)
    2. На каждую — A5 генерит мини-фрагмент (2×4 / 3×3 / строка / столбец) с ровно одной форсированной дедукцией нужного типа
    3. A1 ставит **все** клетки, форсируемые этой дедукцией (1–3 клетки)
    4. ChainDrill: если после ответа появилась новая быстрая дедукция — продолжаем на той же позиции, цепочка пишется как chain-event
    5. После 10-й ситуации — end-of-session summary + кнопка «+10» для продолжения
    6. Кнопка «выйти» доступна всегда, без штрафа на рейтинги
  - **Outcome:** обновлены per-heuristic mastery; A1 видит, что прокачал и где замедлился
  - **Covered by:** R3, R4, R5, R6, R8, R9, R12

- F3. **Hint-as-diagnostic-signal**
  - **Trigger:** A1 нажимает «hint» в обычной партии или drill
  - **Actors:** A1, A2, A3
  - **Steps:**
    1. Лесенка раскрытия, шаг 1: имя дедукции (например «здесь применяется ParityFill»)
    2. Если A1 не нашёл — шаг 2: подсветка клеток-преконда на отдельном «учебном поле» (не на основной доске)
    3. Шаг 3: оптимальный ход
    4. Шаг 4: пояснение «как работает эта дедукция» (если незнакома)
    5. Каждый шаг лесенки = отдельный сигнал слабости в A3 (`hint_step_reached`)
    6. Таймер паузится на время чтения объяснения
  - **Outcome:** A1 разблокирован; A3 получает converging-evidence сигнал «эту дедукцию не вижу даже после имени»
  - **Covered by:** R2, R10, R11

- F4. **Просмотр прогресса (Mastery)**
  - **Trigger:** A1 открывает экран «Mastery» из меню (или из end-of-session summary)
  - **Actors:** A1, A3
  - **Steps:**
    1. Радар-чарт по техникам: percentile-based mastery vs solver-baseline
    2. По каждой технике — медиана/p25/p75 latency, error-rate, hint-rate, FSRS-статус (next due), скользящее окно «последние 7 дней» (настраиваемое)
    3. Список «техник в калибровке» (< ~10–15 событий) — drill для них пока не предлагается
  - **Outcome:** A1 видит, где он силён и слаб, какие техники ещё калибруются
  - **Covered by:** R4, R6, R14

## Requirements

**Каталог дедукций и solver**
- R1. Solver-движок возвращает для любой позиции: список доступных дедукций с тегами, самую дешёвую, какие клетки она форсирует.
- R2. MVP-каталог содержит ~7 именованных техник: `PairCompletion`, `TrioAvoidance`, `ParityFill`, `SignPropagation`, `AdvancedMidLineInference`, `ChainExtension`, `Composite(unknown)` (последний — fallback для дедукций вне словаря, не подаётся в drill).

**Drill-режим**
- R3. Drill-сессия по умолчанию = 10 ситуаций; после 10-й показывается end-of-session summary и кнопка «+10», чтобы продолжить тем же блоком (без полного рестарта сессии).
- R4. На каждую ситуацию: один targeted-heuristic, фрагмент или мини-доска, юзер ставит **все** клетки, форсируемые именно этой дедукцией.
- R5. ChainDrill: если после ответа на доске появилась новая быстрая дедукция, экран не сбрасывается — продолжаем на той же позиции; chain-event пишется в лог.
- R6. Выход из drill — кнопкой «выйти», доступной всегда. Прерывание не понижает рейтинги и не штрафует mastery.
- R29. Recognition drill: помимо production-карточек («найди форсированный ход») drill включает recognition-карточки («есть ли целевой паттерн на доске; если нет — жми „Не нашёл“»). Соотношение по умолчанию 70 production / 30 recognition, калибруется. Без recognition тренажёр учит «всегда что-то кликать», а не «видеть паттерн».
- R30. Three-layer drill difficulty: drill-упражнения структурированы по слоям Pure (только целевой паттерн в фрагменте) → Distracted (тот же фрагмент + правдоподобный шум + похожие, но невалидные паттерны) → Real (полная 6×6, паттерны смешаны). Слой выбирается планировщиком по mastery конкретной техники: новая → Pure, окрепшая → Distracted, mastered → Real.

**Per-heuristic mastery и планировщик**
- R7. На каждую дедукцию хранится: latency-распределение (медиана + p25/p75 + count), error-rate, hint-rate, hint_step_reached counts.
- R8. FSRS-параметры считаются на уровне *техники*, а не отдельной задачи; FSRS-очередь питает выбор drill-карточек.
- R9. Mastery-чарт использует percentile vs solver-baseline (не сырое время).
- R10. Cold-start: техника активируется в drill после ≥ ~10–15 событий калибровки. До этого — drill по ней не предлагается, статистика собирается только из обычных партий.
- R11. Скользящее окно «последние 7 дней» — настраиваемое.

**Hint-as-signal**
- R12. Кнопка hint в любом режиме запускает лесенку: имя → подсветка преконда (на учебном поле) → оптимальный ход → пояснение. Каждый шаг логируется отдельно.
- R13. Таймер паузится на время чтения объяснения.

**Распознавание ходов и контаминация**
- R14. На каждый ход в обычной партии и drill пишется move-event: `{heuristic_tag, latency_ms, was_correct, hint_requested, contaminated_flag}`.
- R15. Ход помечается `contaminated`, если приложение в фоне (`AppLifecycleState.paused`) / телефон неподвижен > N сек / простой > 8 сек. Контаминированные события не учитываются в mastery.
- R16. Таймер скрыт от пользователя; режимы (обычная партия, drill, потенциальные speed-режимы) перемешиваются, чтобы нельзя было оптимизироваться против часов.
- R31. Каждый MoveEvent помечен `mode: propagation | hunt`. Эвристика разметки: ход в радиусе ±1 клетки от предыдущего, или связанный знаком `=`/`×`, и в окне ≤ N сек после предыдущего хода → propagation, иначе hunt. End-of-session показывает долю propagation и p99 hunt latency. N — стартово 5 сек, калибруется по реальной игре. Метрика — индикатор Hunt-веса в общем времени, длинный hunt по конкретному паттерну = слабая узнаваемость.

**Replay-as-Tutor**
- R17. После full-game партии — diff с solver-оптимальным путём, автоматическая генерация drill-карточек для 2–3 самых долгих/неоптимальных пауз; карточки попадают в FSRS-очередь (по умолчанию на завтра).
- R18. UX скруббера показывает только эти 2–3 stuck-moments, а не все ходы.
- R32. Bias-флаг «прошёл мимо очевидного»: если в любой момент партии `ParityFill` с одной пустой клеткой (line_completion в концепт-таксономии) был доступен ≥ 3 секунды и пользователь сделал другой ход — соответствующий ход помечается в end-of-session summary. Это **UX-сигнал, не FSRS-input**: mastery не штрафуется, но пользователь видит, что снова прошёл мимо.

**Генератор уровней**
- R19. Параметрический генератор принимает target-heuristic mix histogram и возвращает уникально-решаемый puzzle с заданным составом.
- R20. Разнообразие обеспечивается двумя механизмами: (e1) target-mix histogram задаётся drill-планировщиком; (e3) хеш прошлых уровней + min-distance threshold отсекает структурно похожие.
- R21. Один генератор обслуживает все режимы: drill-фрагменты, full-game партии, калибровочные пазлы.

**Прогресс UX**
- R22. Прогресс **не показывается** во время игры или drill — никаких индикаторов в углу, никакого «слабая техника сегодня: X».
- R23. End-of-session summary показывается после каждой drill-сессии и каждой full-game партии: что прокачал, где замедлился, какие карточки добавлены в FSRS-очередь.
- R24. Полный экран «Mastery» доступен из меню и из summary: радар-чарт, per-heuristic latency/error/hint, FSRS-статус, список калибрующихся техник.

**Защита от выгорания**
- R25. Soft-nudge: после порога N drill в день (значение по умолчанию определяется при планировании, ориентировочно 30) показывается ненавязчивая плашка «эффект от drill уже падает, отдохни». Ничего не блокируется, очередь не фризится.

**Pluggable architecture**
- R26. Интерфейс `PuzzleKind` (правила-как-constraints, каталог эвристик, рендерер, input-handler) определён так, что Tango — первая реализация. Двигатель тренажёра (телеметрия, mastery, FSRS, drill-планировщик) — puzzle-agnostic. Vertical-slice реализация только для Tango в v1; другие пазлы (Queens, Zip, Binairo) **не делаются** в v1, но интерфейс не должен мешать их добавлению.

**Локальность и отсутствие рекламы**
- R27. Всё хранение — локально (sqlite), без сервера, без облачной синхронизации, без аккаунтов.
- R28. Без рекламы, без analytics, без telemetry в третьи стороны.

**UX naming и таксономия**
- R33. UI использует человекочитаемые имена паттернов («Заполнение линии», «Запрет тройки», «Краевая ловушка 1–5», «Цепочка знаков», ...) через `HeuristicDescriptor.displayName`, а не движковые `tagId` (`PairCompletion`, `AdvancedMidLineInference`, ...). Mapping зафиксирован в [`docs/tango_trainer_concept_addendum.md`](../tango_trainer_concept_addendum.md). Движковые `tagId` не меняются — они зафиксированы в БД и в коде Phase A/B; переименование = миграция, что v1 не делает.

## Acceptance Examples

- AE1. **Covers R3, R6.** Игрок начал drill, сделал 3 из 10 ситуаций, нажал «выйти». Сессия закрывается, частичный summary показан, mastery по сделанным 3-м обновлено, остальные 7 — не штрафуют ничего. FSRS-очередь не наказывается.

- AE2. **Covers R12, R13.** В обычной партии игрок жмёт hint. Видит «здесь ParityFill», ничего не делает 5 сек, жмёт снова — подсвечиваются клетки-преконда на учебном поле, основная доска нетронута. Жмёт ещё — оптимальный ход. Жмёт ещё — текстовое пояснение. Логируется 4 события: `hint_step_reached: name`, `precond`, `move`, `explanation`. Таймер всё это время на паузе.

- AE3. **Covers R10.** Техника `AdvancedMidLineInference` имеет 7 событий (< 15). Игрок открывает drill — карточек на эту технику нет, она помечена «в калибровке» в Mastery-экране. Игрок продолжает играть обычные партии — на 16-м событии техника становится доступной для drill.

- AE4. **Covers R5.** Drill-карточка показывает форсированный `ParityFill`. Игрок ставит обе клетки. После этого на доске стало видно `SignPropagation` — экран не очищается, новая дедукция доступна сейчас на этой же позиции. Игрок ставит её клетку. Записан chain-event `ParityFill → SignPropagation`. Карточка засчитана, переход к следующей.

- AE5. **Covers R15.** Игрок свернул приложение на 12 секунд во время хода в drill. Возвращается, ставит клетку. Move-event пишется с `contaminated_flag: true`, mastery по этой технике не обновляется этим событием.

- AE6. **Covers R17.** Игрок закончил full-game партию за 4 мин. Solver-baseline = 2:30. Из 28 ходов 3 были существенно медленнее baseline. Система автоматически создаёт 3 drill-карточки на соответствующие эвристики, ставит их в FSRS-очередь на завтра. Игрок видит summary: «3 карточки добавлены».

- AE7. **Covers R22, R23, R24.** Во время drill ни один индикатор «слабая техника» не показывается. После drill — summary: «прокачал ParityFill (медиана −0.4с), замедлился на TrioAvoidance». Игрок жмёт «Подробнее» → открывается Mastery-экран с радар-чартом.

- AE8. **Covers R25.** Игрок сделал 30 drill-карточек подряд. На 31-й перед стартом — плашка «эффект от drill уже падает, отдохни. Продолжить?» с кнопками «продолжить» / «закрыть». Кнопка «продолжить» запускает следующую карточку.

- AE9. **Covers R29, R30.** Drill-сессия по технике `TrioAvoidance` стартует на Слое 1 (Pure): фрагмент 2×4 с явно стоящей парой `SS_`. Карточка №1 — production: игрок ставит moon в третью клетку → correct. Карточка №2 — recognition: фрагмент того же размера, но без целевого паттерна (все остатки определены другими эвристиками). Игрок жмёт «Не нашёл» → `recognition_correct_reject` записан. Карточка №3 — recognition с паттерном, игрок ошибается «Не нашёл» → `recognition_false_alarm`, FSRS-rating Hard. После сессии mastery TrioAvoidance: 1 production-hit, 1 correct_reject, 1 false_alarm. По мере роста mastery следующая сессия по этой технике стартует на Слое 2 (Distracted) с шумом вокруг.

- AE10. **Covers R31, R32.** Full-game партия 6×6, 28 ходов, 4:12. После хода ParityFill в строке 3 следующий ход — клетка через знак `=` в строке 5 (вне радиуса propagation, > 5 сек паузы). Этот ход помечается `mode=hunt`. Summary показывает: «доля propagation 60%, p99 hunt latency 9.2 сек, самый медленный паттерн в hunt-mode: edge_1_5_trap». Дополнительно флаг bias: на ходе #14 был доступен `ParityFill` с одной пустой клеткой 4.1 сек до того, как пользователь сделал другой ход — отображается «снова прошёл мимо очевидного». Mastery по ParityFill этим bias-сигналом не штрафуется (R32).

## Success Criteria

- Через 4–6 недель ежедневного использования игрок видит измеримый рост mastery (percentile vs solver-baseline) хотя бы по 3 техникам, и хотя бы одна слабая техника по началу больше не помечается как «слабая».
- В обычной партии латентность хода по знакомой технике снижается заметнее, чем по технике, которая в drill не подаётся (контрольная группа техник внутри одного игрока).
- Hint-rate по конкретной технике снижается после регулярного drill этой техники.
- Игрок не ощущает «давления тренажёра»: drill — желанная сессия, выход всегда без штрафа, прогресс не мозолит глаза.
- Downstream-handoff: `/ce-plan` может начать архитектурный план без новых продуктовых решений; реализатор не должен изобретать UX или scope.

## Scope Boundaries

- Никакой облачной синхронизации, никаких аккаунтов, никаких leaderboards.
- Никакой рекламы, никаких сторонних analytics.
- Никакого «агента, который играет за тебя» — только подсказки по запросу и drill-предложения.
- Никаких многоходовых/машинных дедукций в каталоге MVP — `Composite(unknown)` логируется, но в drill не подаётся (расширение каталога — по сигналу накопления случаев).
- Никакого специального «диагностического онбординга» — cold-start = обычные партии.
- Никаких дополнительных пазлов (Queens, Zip, Binairo) в v1 — только Tango. Интерфейс `PuzzleKind` готов, но реализация только одна.
- Никакого hard-cap или freeze-очереди — только soft-nudge.
- Никаких индикаторов прогресса в игровом процессе — только end-of-session summary и отдельный Mastery-экран.
- Eye-tracking / dwell-time / heatmap взгляда — отложено как v2+.
- Watch-приложение, voice-input, multi-solution puzzles, variable grid size — все отложены.
- LinkedIn-mirror mode (тренироваться на сегодняшнем LinkedIn-пазле) — v2.

## Key Decisions

- **Drill-сессия = база 10 + кнопка «+10»**: компромисс между ясным брейкпоинтом для выхода и «играю пока хочу». Фиксированная длина даёт честный summary; «+10» снимает фрустрацию принудительной остановки.
- **Прогресс = end-of-session + отдельный экран Mastery, без индикаторов в игре**: уважаем поток игры, не превращаем приложение в дашборд. Mastery-экран — для рефлексии, summary — для микрообратной связи.
- **Soft-nudge вместо hard-cap**: это личный инструмент, взрослый режим. FSRS сам разреживает очередь, плашка после N drill достаточна.
- **MVP только Tango, но интерфейс PuzzleKind заложен**: один пазл — фокус, чёткий feedback loop. Plug-API нужен сейчас, чтобы будущие добавления не требовали переписывать движок.
- **Mastery = percentile vs solver-baseline**: сырое время некорректно (Nature 2015 RT pitfall). Solver-baseline нормализует сложность позиции; percentile делает прогресс читаемым.
- **Converging-evidence для слабости техники**: latency сам по себе не диагностичен. Слабость = latency + (ошибка ИЛИ hint ИЛИ повтор через эпизоды).
