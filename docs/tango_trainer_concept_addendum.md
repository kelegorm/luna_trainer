# Tango Trainer — дополнения к концепт-документу

> **Дополняет:** [`tango_trainer_concept.md`](./tango_trainer_concept.md)
> **Применяется к:** [`brainstorms/2026-05-01-tango-trainer-requirements.md`](./brainstorms/2026-05-01-tango-trainer-requirements.md), [`plans/2026-05-01-001-feat-tango-trainer-mvp-plan.md`](./plans/2026-05-01-001-feat-tango-trainer-mvp-plan.md)
> **Статус:** ideation для Phase D (UX). Phase A (foundation) и Phase B (solver + generator) уже зафиксированы в коде на ветке `feat/phase-b-tango-solver`.

## Контекст

Концепт-документ описывает **педагогическую** модель тренажёра: какие паттерны существуют, как их называть для игрока, как ramp-up'ить сложность, зачем нужны negative examples. План v1 описывает **инженерную** модель: solver, FSRS, telemetry, contamination. Эти два среза не противоречат — они дополняют друг друга, и расхождения в наименованиях дедукций связаны с тем, что план называет техники по их **алгоритмическому действию**, а концепт — по тому, **как их видит человек**.

Этот документ:

1. Сводит таксономии в одну таблицу (concept ↔ движок ↔ display).
2. Перечисляет идеи концепт-документа, которые стоит включить в Phase D.
3. Объясняет, что мы **не** берём (или берём в модифицированном виде) и почему.
4. Поднимает открытые вопросы для следующего планирования.

## Таксономия: одно и то же, разный угол

| Концепт-имя | Движковое имя (`Heuristic.tagId`) | UI `displayName` (предлагается) |
|---|---|---|
| `line_completion` (5/6 → 6-я форсирована) | `ParityFill` (частный случай: 1 пустая) | «Заполнение линии» |
| `no_three_pair_left` (`SS_`) | `TrioAvoidance` | «Запрет тройки (пара)» |
| `no_three_pair_right` (`_SS`) | `TrioAvoidance` | «Запрет тройки (пара)» |
| `no_three_gap` (`S_S`) | `PairCompletion` | «Запрет тройки (зазор)» |
| `balance` (3 одного → остальные другого) | `ParityFill` | «Баланс линии» |
| `edge_1_5_trap` (`A___A_`) | `AdvancedMidLineInference` | «Краевая ловушка 1–5» |
| `edge_2_6_trap` (`_A___A`) | `AdvancedMidLineInference` | «Краевая ловушка 2–6» |
| `relation_same` (`X = Y`, X известна) | `SignPropagation` | «Знак равенства» |
| `relation_opposite` (`X × Y`, X известна) | `SignPropagation` | «Знак неравенства» |
| `relation` chain (`X = Y × Z = W`) | N×`SignPropagation` подряд (на уровне solver-trace), мета-кластер `ChainExtension` | «Цепочка знаков» |

**9 строк из 10 семантически идентичны** — игрок и solver описывают одно и то же действие на разном уровне детализации (`no_three` группирует то, что у нас разнесено на `TrioAvoidance` + `PairCompletion`). Это решается через `HeuristicDescriptor.displayName`, без изменения движка.

### Один реальный нюанс — chain через `=` / `×`

Игрок распознаёт `X = Y × Z = W` как **один** паттерн узнавания. Solver декомпозирует это в N одно-шаговых `SignPropagation`-дедукций. Для UX-узнаваемости важно, чтобы:

- Hint-ladder группировал последовательные `SignPropagation` по одной chain в **одно** hint-сообщение, а не переключался N раз.
- End-of-session summary показывал «решена цепочка из 4 знаков», а не «4 отдельных дедукции».

Это **задача рендеринга / drill-Bloc**, не движка. `ChainExtension` уже маркирует второе и последующие звенья — этим тегом можно пользоваться для группировки.

## Идеи концепт-документа для Phase D

### 1. Negative examples + кнопка «Не нашёл»

**Что есть в плане:** drill «найди следующий форсированный ход» (production-mode).

**Что добавляет концепт:** 70% упражнений с паттерном, 30% — без, с правильным ответом «не нашёл». Без negative examples тренажёр учит «всегда что-то кликать», а не «видеть паттерн».

**Предложение для Phase D:**

- Новый drill-режим **recognition** рядом с production. Производитель — тот же level-generator (фрагмент 2×4 / 3×3 + опционально пустой), но 30% задач генерятся **без** целевого паттерна (все клетки уже определены простыми эвристиками без триггера для целевой).
- В `move_events_table` добавить поле `event_kind: production | recognition_hit | recognition_correct_reject | recognition_false_alarm`.
- FSRS-rating маппится: `recognition_hit` или `production` правильно → Good; `recognition_false_alarm` → Hard или Again.

**Затронутые требования:** R3, R4, R5, R6 (drill mechanics) — расширяются, не заменяются.

### 2. Трёхслойная сложность Pure → Distracted → Real

**Что есть в плане:** drill-фрагменты (2×4, 3×3) + полная партия как разные сущности; нет явной градации шума.

**Что добавляет концепт:**

- **Слой 1 (Pure):** только целевой паттерн в фрагменте, без шума. Изоляция.
- **Слой 2 (Distracted):** тот же фрагмент + правдоподобный шум (другие частично-заполненные строки), целевой паттерн один.
- **Слой 3 (Real):** полная 6×6, паттерны смешаны.

**Предложение для Phase D:**

- `BoardShape` уже различает 6×6 / 2×4 / 3×3 (R26 готов). Нужен новый признак `DrillLayer { pure, distracted, real }` в `DrillSelector` и при генерации.
- Слой 1 ↔ генератор с `target_mix={target: 1.0}` на фрагменте + minimum constraints.
- Слой 2 ↔ тот же фрагмент, но с дополнительным шумом (1–2 «отвлекающих» полу-заполненных линии вокруг).
- Слой 3 ↔ существующий full-game режим.
- FSRS должна учитывать слой: переход 2 → 3 это «advance», а не очередное повторение.

**Затронутые требования:** R3, R4 (drill structure) — добавляется новое измерение.

### 3. Propagation vs Hunt — отдельная телеметрическая метрика

**Что добавляет концепт:** различение «доделать локальную цепочку после хода» (Propagation) vs «глобальный скан» (Hunt). У сильных игроков Propagation быстрее Hunt; доля времени в Hunt — индикатор слабых паттернов.

**Предложение:**

- В `move_events_table` добавить поле `mode: propagation | hunt`. Эвристика разметки: ход в радиусе ±1 клетки от предыдущего, или связанный с предыдущим знаком `=`/`×`, и в течение ≤ N секунд после предыдущего → propagation; иначе hunt. N — стартовое значение 5 секунд, калибруется (Open Question).
- Сводка в end-of-session: «доля propagation = X%, p99 hunt latency = Y с — самый медленный паттерн в hunt-mode: Z».

**Затронутые требования:** R14, R15 (telemetry) — расширяется.

### 4. Когнитивная пометка: `line_completion` склонен игнорироваться

**Что добавляет концепт:** наблюдение, что мозг пропускает `line_completion` (5/6) в поиске «более интересных» дедукций.

**Предложение:** в end-of-session summary флагать ходы, где `line_completion`-эквивалент (`ParityFill` с 1 пустой клеткой) был доступен ≥ 3 секунды **до** того, как пользователь его сделал, и при этом пользователь сделал что-то другое. Это разовый bias-сигнал, **не** FSRS-input — не повышаем weakness техники, но показываем сообщение «ты снова прошёл мимо очевидного». Полезно как UX-feedback, дёшево в реализации.

**Затронутые требования:** R17, R18 (replay-as-tutor) — расширяется.

### 5. Пользовательские имена паттернов в UI

**Что есть в плане:** `HeuristicDescriptor.displayName` уже зафиксирован в `lib/puzzles/tango/tango_puzzle_kind.dart`, но значения — машинные («Pair completion», «Trio avoidance»).

**Что добавляет концепт:** более узнаваемые человеком имена (см. таблицу выше).

**Предложение:** ровно один Edit в `tango_puzzle_kind.dart` под значения из таблицы. Движковые имена остаются неизменными — там в БД, миграция не нужна. **Один открытый вопрос:** `edge_1_5` и `edge_2_6` — подвиды `AdvancedMidLineInference`, и для drill хочется тренировать их по отдельности. Это решается под-классификацией (см. Open Questions).

## Что **не** берём (или берём в модифицированном виде)

### Penalty seconds (+1 за FP, +2 за miss) — не заменяют FSRS

Концепт предлагает **секундные штрафы** как основу скоринга. План уже использует FSRS + EWMA + shrinkage — это даёт долгосрочное планирование повторений, чего штраф-секунды не дают. Это две разные системы измерения, и они конфликтуют как «main scoring».

**Что берём:** штраф-секунды используем **только** как UX-сигнал в end-of-session summary («ты потерял 4.2 секунды на FP»), не как input в FSRS-rating и не как сигнал в mastery-update. Mastery продолжает обновляться через RT + correctness, как описано в плане. Если за 2 недели live-use окажется, что штраф-метрика лучше коррелирует со скоростью реального solve — переоценим в follow-up планировании.

### Полная замена таксономии — слишком дорого после Phase B

Имена концепта (`no_three_gap`, `edge_1_5_trap`, ...) лучше для игрока, но переименование всех `tagId` в коде — большой diff (`lib/puzzles/tango/solver/`, `tango_puzzle_kind.dart`, тесты, фикстуры) и **миграция данных** в `mastery_state` / `fsrs_cards` / `move_events` (после первого запуска), не давая функциональной разницы.

**Что берём:** mapping через `displayName` решает 100% UX-задачи без ребейза кода и без миграций. Если когда-нибудь окажется, что `tagId` тоже надо переименовать (например, для open-source taxonomy v1), это будет отдельная миграция в более позднем фазе.

## Открытые вопросы для Phase D

1. **Под-классификация `AdvancedMidLineInference` на `edge_1_5` / `edge_2_6` / прочее.** Для отдельной тренировки каждой под-формы нужен либо разбиение на 2–3 sibling-эвристики, либо тег внутри `TangoDeduction` без изменения движкового интерфейса. Оценка: 2 часа на разбиение и обновление тестов U5.

2. **Группировка цепочки `SignPropagation` в один UI-event.** Чисто рендерная задача в drill-Bloc; группировка по `ChainExtension`-meta уже есть в solver-output.

3. **Радиус и таймаут propagation-метки.** Стартово: ±1 клетка / связь одним знаком / ≤ 5 секунд после предыдущего хода. Финальные значения после 2 недель live-use.

4. **Recognition drill ratio (70/30).** Стартовое из концепта; калибруется по фактическому false-alarm rate в первые недели использования.

5. **Маппинг `recognition_*` событий в FSRS-rating.** Стартовая модель в этом документе (hit→Good, false_alarm→Hard, correct_reject→Good). Альтернатива: false_alarm не маппить вообще (drill-уровень не апдейтит mastery, только UX feedback). Решение — после первой пары недель данных.

---

## Итог

Концепт-документ **не заменяет** план — он накладывается сверху как педагогический и UX-слой. Конкретно для Phase D из него стоит взять: recognition drill с negative examples (R3-R6 расширение), три слоя сложности (R3-R4), propagation/hunt метрику (R14-R15), bias-флаги в summary (R17-R18), и человекочитаемые `displayName`.

Penalty seconds оставляем как UX-сигнал, не как scoring-вход. Таксономию в коде не трогаем — mapping через `displayName` решает UX-задачу без миграций.
